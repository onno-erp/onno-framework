# onec → onno cutover runbook (maintainer)

The one-time process to ship the rename and roll it out. Order: **finalize the branch →
verify CI/publishing → merge to main → release a version → migrate consumers one by one,
starting with `onec-enterprise`.**

> Companion docs (consumer-facing, already written):
> - [`onec-to-onno.md`](onec-to-onno.md) — what each consumer app must change.
> - [`onec-to-onno.sql`](onec-to-onno.sql) — the per-database table/index rename.

Current state: the full rename is staged on branch **`rename/onno`** (not merged). Build +
tests + an example-app runtime + DB-migration are all verified green on that branch.

---

## Phase 0 — Prerequisites & decisions (do before touching the branch)

- [ ] **Maven Central namespace** `su.onno` is *Verified* on central.sonatype.com.
      (Confirmed.) Releases are immutable — a bad version is permanent.
- [ ] **The GitHub secret `MAVEN_CENTRAL_USERNAME` / `MAVEN_CENTRAL_PASSWORD` is a User Token
      generated by the Sonatype account that owns the verified `su.onno` namespace.** If
      `su.onno` was verified under a different account than the one currently in secrets,
      regenerate the token under the right account (Account → Generate User Token) and update
      both repo secrets. A namespace/account mismatch fails publish with `403 not authorized`.
- [ ] **GPG signing secrets** still valid: `SIGNING_IN_MEMORY_KEY`, `SIGNING_IN_MEMORY_KEY_PASSWORD`.
      (Unchanged by the rename; Central requires signed artifacts.)
- [ ] **Pick the version.** New coordinates (`su.onno:onno-*`) are brand-new artifacts on
      Central — the old `io.github.onec-erp:onec-*` line stays at its last version forever.
      Recommended: **`1.0.0`** to mark the namespace milestone and the breaking change. (The
      version is driven entirely by the git tag / `-PreleaseVersion`; nothing is hardcoded
      except the desktop plugin — see Phase 2.)

## Phase 1 — Finalize the rename branch

### 1a. Rebase onto the latest main, then RE-SWEEP

`rename/onno` was cut from main at an older commit. Any commits merged to main since then
(features, fixes) reintroduce `onec` strings that the original sweep never saw.

```bash
git checkout rename/onno
git fetch origin
git rebase origin/main           # resolve conflicts favoring the renamed side
```

Then re-run the **full ordered rename pass** over any newly-introduced occurrences
(Appendix A) and the post-pass fixes, then the **residual check** (Appendix B). The residual
check MUST come back clean except the known `OneColumn` false positive.

### 1b. Green-build gate (must all pass)

```bash
./gradlew clean build                 # all modules + tests (Java + 36 frontend tests)
./gradlew generateConfigDocs checkConfigDocs   # CONFIGURATION.md regenerated & in sync
git grep -nIE '\bonec|Onec|onec_|com\.onec|io\.github\.onec-erp'   # only OneColumn allowed
git ls-files | grep -i onec           # must be empty (no onec in any path)
```

### 1c. Runtime smoke (optional but recommended)

```bash
rm -f data/*.mv.db                     # fresh H2 so it re-seeds with onno_ tables
java -jar example/build/libs/example-0.1.0-SNAPSHOT.jar
# expect: "Started ExampleApp", /actuator/health=200, onno_ tables, no onec_
```

## Phase 2 — CI / publishing readiness

- [ ] **Framework artifacts**: the 10 published modules (`onno-framework`, `*-starter`) use the
      vanniktech plugin with `group = "su.onno"`; the tag workflow runs `./gradlew
      publishToMavenCentral`. No workflow edit needed — verify with a **local dry run**:
      ```bash
      ./gradlew publishToMavenLocal -PreleaseVersion=1.0.0-rc1
      ls ~/.m2/repository/su/onno/          # expect onno-framework, onno-*-starter dirs
      ```
- [ ] **Desktop Gradle plugin** (`onno-desktop-gradle-plugin`): it is a separate `includeBuild`,
      NOT part of the tag workflow's `publishToMavenCentral`, and still pins `version = "0.1.0"`.
      Decide its release path before consumers can `id("su.onno.desktop")`:
      - Align its version with the release (replace the hardcoded `0.1.0`), and
      - publish it (Gradle Plugin Portal, or Central as a plugin-marker) as its own step.
      If desktop packaging isn't part of this release's surface, leave it local-only and say so
      in the release notes.
- [ ] **Docs site**: `docs.yml` deploys to Pages → `docs.onno.su` (CNAME unchanged). No action.
- [ ] **`ci.yml`**: PR build/test only; namespace-agnostic. No action.

## Phase 3 — Merge to main

- [ ] Open a PR `rename/onno → main`. Expect ~634 renames + ~660 edits.
- [ ] **Merge with a merge commit (`--no-ff`), do NOT squash.** Squashing collapses every
      `git mv` into one blob and destroys rename detection / `git log --follow` history across
      the cutover. A true merge preserves per-file lineage.
- [ ] After merge, confirm main builds: `git checkout main && git pull && ./gradlew clean build`.

## Phase 4 — Release the version

Maven Central is immutable — **dry-run with a release candidate first.**

1. **RC dry run** (validates the `su.onno` namespace + signing end-to-end on the real Portal):
   ```bash
   git tag v1.0.0-rc1 && git push origin v1.0.0-rc1
   ```
   The workflow detects `-rc1` → `prerelease=true`. Watch the `Publish to Maven Central` step;
   then confirm the deployment validates at https://central.sonatype.com (and, with
   `SONATYPE_AUTOMATIC_RELEASE=true`, that `su.onno:onno-framework:1.0.0-rc1` resolves from
   `repo1.maven.org` within ~15–30 min).
2. **Final release**:
   ```bash
   git tag v1.0.0 && git push origin v1.0.0
   ```
   The workflow runs `clean check`, publishes all 10 modules under `su.onno`, and creates the
   GitHub release with generated notes.
3. **Verify**: each `su.onno:onno-*:1.0.0` is resolvable; the
   [Central namespace page](https://central.sonatype.com/namespace/su.onno) lists them.
4. **Release notes** must state: new coordinates `su.onno:onno-*`, new package root `su.onno.*`,
   config prefix `onno.*`, and link [`onec-to-onno.md`](onec-to-onno.md) +
   [`onec-to-onno.sql`](onec-to-onno.sql).

## Phase 5 — Roll out to consumers, `onec-enterprise` first

Each consumer repo is its own rename, mechanically identical to what was done here. Do them
**one at a time, fully verified, before the next.** Start with `onec-enterprise` (it depends on
the framework and is itself a publisher).

### Per-consumer playbook (template)

1. **Branch** `rename/onno` in the consumer repo.
2. **Bump the framework dependency** to the new coordinates/version:
   `io.github.onec-erp:onec-*:<old>` → `su.onno:onno-*:1.0.0`, and the plugin
   `com.onec.desktop` → `su.onno.desktop` if used.
3. **Run the rename pass on the consumer's OWN code** (Appendix A) — its packages
   (`com.onec.enterprise` → `su.onno.enterprise`), class prefixes, and any `onec.*` config it
   defines. Apply the same post-pass fixes (Appendix A) and residual check (Appendix B).
4. **Config / env / metrics**: apply [`onec-to-onno.md`](onec-to-onno.md) to the consumer's
   deployment — `ONEC_*`→`ONNO_*` env vars, `onec.*` props, `-Donec.*`, logging levels,
   and Grafana/Prometheus `onec.operation.*` → `onno.operation.*`.
5. **Database**: run [`onec-to-onno.sql`](onec-to-onno.sql) against each of that consumer's
   environments. **Plus any consumer-specific `onec_*` tables** — e.g. `onec-enterprise` has its
   own (`onec_hospedajes_comunicacion`, etc.); enumerate them with
   `git grep -ohE 'onec_[a-z_]+'` in that repo and add matching `ALTER TABLE IF EXISTS … RENAME`
   lines. Snapshot self-heals (same diff-engine behavior as the framework).
6. **Consumer CI/publishing** (if it publishes, like the enterprise connectors): same as Phase 2 —
   set its `group`/coordinates to `su.onno`(.enterprise), confirm its Central/registry account
   owns the namespace.
7. **Verify**: `./gradlew clean build`; residual grep clean; boot against a migrated DB and
   confirm 0 schema changes + data intact (numbering counters preserved); then merge & release
   that consumer.

### Suggested order
`onec-enterprise` (guesty / hospedajes / tochka connectors) → any internal apps that consume the
framework directly → external/community consumers (give them the release notes + the two
consumer docs; they self-migrate).

## Appendix A — the reusable rename pass

Run from the repo root, on a branch, after `git mv`-ing module dirs, package dirs
(`com/onec` → `su/onno`), and `Onec*.java` → `Onno*.java`. Operates on tracked text files;
**order matters** (specific → general):

```bash
while IFS= read -r -d '' f; do
  [ -f "$f" ] || continue            # skip symlinks/dirs
  grep -Iq . "$f" || continue        # skip binary/empty (see post-pass note!)
  perl -i -pe '
    s/\Qio.github.onec-erp\E/su.onno/g;   # groupId (before com/onec, before generic)
    s/com\.onec/su.onno/g;                # package refs
    s{com/onec}{su/onno}g;                # package PATH strings (also hits URLs — see below)
    s/Onec/Onno/g;                        # class names + camelCase humps
    s/\bonec/onno/g;                      # word-boundary: onec-*, onec.*, onec_*, css, etc.
  ' "$f"
done < <(git ls-files -z)
```

**Mandatory post-pass fixes** (each caught a real bug here):

```bash
# 1. uppercase tokens, underscore-preceded tokens, and the smushed markdown anchor
while IFS= read -r -d '' f; do [ -f "$f" ] || continue; grep -Iq . "$f" || continue
  perl -i -pe 's/ONEC/ONNO/g; s/_onec/_onno/g; s/comonecdesktop/suonnodesktop/g;' "$f"
done < <(git ls-files -z)

# 2. the `com/onec` rule also rewrites `github.com/onec-erp` → `github.su/...`; restore the host
while IFS= read -r -d '' f; do [ -f "$f" ] || continue; grep -Iq . "$f" || continue
  perl -i -pe 's/github\.su/github.com/g' "$f"
done < <(git ls-files -z)

# 3. files with a non-ASCII byte are flagged "binary" by grep -I and SKIPPED above.
#    Find and fix them explicitly (here it was one CSV-import .java):
git ls-files -z | xargs -0 grep -ailZ 'onec' 2>/dev/null | tr '\0' '\n'
#    → for each (except a genuine false positive), run the full perl chain on it directly.

# 4. resource DIRECTORIES with onec in a PARENT segment are not caught by basename file renames.
#    The *reference* string gets renamed but the dir doesn't → runtime "resource not found".
git ls-files | grep -i 'onec'        # rename any such dirs/files with `git mv`
```

## Appendix B — residual verification (must be clean)

```bash
git grep -nIE '\bonec|Onec|onec_|com\.onec|io\.github\.onec-erp'   # only OneColumn-style false positives allowed
git ls-files | grep -i onec                                        # empty: no onec in any tracked path
git grep -nE '\.su/' | grep -v 'onno\.su'                          # empty: no github.su / mangled hosts
```

Known acceptable hit: an identifier containing the literal words "one column" / "one c…"
(e.g. `multiSelectFoldsValuesForOneColumn…`) — `OneC` ≠ `Onec`, leave it.

---

## Appendix C — safety notes

- **Maven Central is immutable.** Never reuse a version number. Guard releases at the tag
  (only push `v*.*.*` when you mean it). Use `-rcN` tags to rehearse.
- **DB migration is one-way per environment.** Back up before running `onec-to-onno.sql`.
  Rollback = restore the backup (or reverse the renames + restore the `onec_` snapshot).
- **Don't run mixed old/new cluster nodes**: the Postgres NOTIFY channel default changed
  (`onec_cluster_events` → `onno_cluster_events`), so a half-upgraded cluster splits its bus.
- **Old artifacts remain consumable.** Existing apps pinned to `io.github.onec-erp:onec-*`
  keep resolving from Central — migration is opt-in per consumer, at your pace.
